## v2.3.0 - 2026-05-23

### 优化

- 登录与账户接管链路修复：钱包地址登录必须增加 challenge 签名校验，禁止“仅凭地址登录”
- 对象级授权（IDOR/BOLA）加固：仓位、止盈止损、银行卡、后台审核/调账/KYC 等接口统一增加“资源归属校验 + 角色范围校验”
- 后台高危越权封堵：充值/提现审核、调账、改密、拉黑、代理归属等操作增加强校验（团队边界、审批链、不可越权）

- 2FA 密钥泄露处置：下线匿名可访问二维码/密钥接口，轮换历史密钥
- 资金回调验签强制化：充值/提现回调增加签名校验、时间窗校验、nonce 防重放，失败即拒绝
- 资金状态机幂等化：充值、提现、退款、结算、划转全链路引入幂等键与状态 CAS，避免重复入账/退款/派奖
- 事务一致性改造：内部划转、结算、审核资金动作改为单事务或可靠补偿事务，防止中间态失衡
- 密码与凭据治理：移除默认弱口令（如 `123456`），改密接口强制绑定当前登录主体
- 敏感日志脱敏：禁止打印 raw body、sign、证件号、手机号、地址、密钥，接入统一脱敏中间件
- 前端高危操作防重：提现/下单/审核/保存做前后端双重防重复（按钮锁 + 服务端幂等）
- URL 传敏整改：敏感参数（提现、2FA、验证码、密钥等）不得通过 URL 传递
- 密钥管理体系升级：移除配置/代码中的硬编码密钥，迁移至密钥管理服务
- XSS 治理：公告/帮助中心/站内信等富文本入口启用内容净化、输出编码与 CSP
- SSRF 与不安全 HTTP 工具治理：目标地址白名单、禁用 trust-all SSL、补齐证书与主机名校验
- 依赖治理与 SBOM：升级高危依赖，移除高风险 `system-scope` 本地 jar，建立组件清单
- 稳定性优化：邀请码 `while(true)` 改为有限重试 + 兜底方案，避免高并发性能衰减
- 环境配置隔离：防止 dev 配置串用到生产，增加启动时环境校验
- Druid 监控台公网暴露与默认口令风险是否仍存在
- 富文本存储型 XSS 是否可稳定触发
- SSRF/LFI/MITM 工具链是否可形成实际可利用路径
- 依赖漏洞在当前运行路径是否可达（需可达性验证）

## 8.3 充值方式

本版本的充值方式已经改为对接到三方优盾钱包，之前是手动充值上分方式，太掉价了。

至于自建钱包节点，没技术运维保证下，成本更高（主要是安全）。

### 8.3.1 关于充值地址

![image-20260308121506618](./assets/image-20260308121506618.png)



**充值业务流程图**

```mermaid
sequenceDiagram
    autonumber
    participant User as 用户 (前端)
    participant Exchange as 交易所 (后端)
    participant Udun as 优盾钱包 (网关)
    participant Chain as 区块链网络

    User->>Exchange: 1. 请求充值，选择币种/网络
    Exchange->>Exchange: 2. 查询本地是否已有绑定地址
    
    opt 若无可用地址
        Exchange->>Udun: 3. 调用 [获取新地址] API
        Udun-->>Exchange: 4. 返回新地址
        Exchange->>Exchange: 5. 将新地址与该用户ID绑定
    end
    
    Exchange-->>User: 6. 展示用户的专属充值地址
    
    User->>Chain: 7. 用户向该地址发起链上转账
    Chain->>Udun: 8. 优盾节点监控到交易上链
    Udun->>Udun: 9. 等待达到设定的安全确认数
    
    Udun->>Exchange: 10. 发起 POST 异步回调 [充值成功] (含TXID/金额)
    Exchange->>Exchange: 11. 验签 (防伪造) & 校验TXID (防重放)
    Exchange->>Exchange: 12. 给对应用户加钱，更新充值订单状态
    Exchange-->>User: 13. 充值到账，可用余额增加

```

**提现业务流程图**

```mermaid
sequenceDiagram
    autonumber
    participant User as 用户 (前端)
    participant Exchange as 交易所 (后端)
    participant Udun as 优盾钱包 (网关)
    participant Chain as 区块链网络

    User->>Exchange: 1. 发起提现申请 (目标地址/金额/验证码)
    Exchange->>Exchange: 2. 检查账户可用余额是否充足
    Exchange->>Exchange: 3. 扣除可用余额，增加冻结余额 (资金冻结)
    Exchange->>Exchange: 4. 内部风控审核 (黑名单/大额拦截等)
    
    opt 审核通过
        Exchange->>Udun: 5. 调用 [提交提现] API (携带唯一业务单号)
        Udun->>Udun: 6. 优盾内部验签、风控、构建并签名交易
        Udun->>Chain: 7. 广播交易至区块链网络
        Udun-->>Exchange: 8. 同步返回交易Hash (TXID)
        Exchange-->>User: 9. 提现状态更新为 "处理中"
    end
    
    Chain->>Udun: 10. 优盾节点持续监控该笔交易的链上状态
    Udun->>Exchange: 11. 发起 POST 异步回调 [提现最终结果]
    
    Exchange->>Exchange: 12. 验证回调请求的签名
    
    alt 若回调状态为: 提现成功
        Exchange->>Exchange: 13. 扣除已冻结的余额，订单状态改为 "成功"
    else 若回调状态为: 提现失败 (如链上失败/被退回)
        Exchange->>Exchange: 14. 解冻资金，退回用户的可用余额，订单"失败"
    end
    
    Exchange-->>User: 15. 通知提现最终结果
```





## 8.4 注册登录

三种方式：

| 账号+密码      | 无需任何认证，直接注册成功             |
| -------------- | -------------------------------------- |
| 邮箱+密码      | 需要配置STMP ，例如谷歌邮箱            |
| TP钱包授权登录 | web3 或者小狐狸钱包授权，几乎没启用    |
| 手机号         | 需要重新对接短信服务商，太多小厂不稳定 |



## 8.5 关于客服

交易所没有提供客服功能，因客服系统的特殊串流方式，和交易所集成，会有很大的安全隐患，所以客服可以对接三方，或者自建。
